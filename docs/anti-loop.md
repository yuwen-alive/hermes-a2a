1|# 防死循环机制
2|
3|> 多重保障确保 bot 间对话不会陷入无限循环
4|
5|## 1. 问题背景
6|
7|当两个 AI bot 在同一个群聊中互相 @mention 对话时，天然存在死循环风险：
8|
9|- Bot A 回复 Bot B → Bot B 认为需要回复 Bot A → Bot A 又回复...
10|- 两个 bot 对同一问题反复表达相同观点，无法收敛
11|- 一方发送的消息触发另一方的回复，形成无限乒乓
12|
13|本机制通过**五层防护**彻底解决这个问题。
14|
15|---
16|
17|## 2. 第一层：信号标记（协议级）
18|
19|### 机制
20|
21|每条消息必须携带信号标记，明确告知对方是否需要回复。
22|
23|```
24|[CONTINUE]  → 需要回复
25|[DONE]      → 不需要回复（任务完成）
26|[NEEDS_HUMAN] → 不需要回复（升级给人类）
27|[INFO]      → 不需要回复（纯通知）
28|```
29|
30|### 规则
31|
32|| 收到的信号 | 必须执行的操作 |
33||-----------|-------------|
34|| `[CONTINUE]` | 回复（可以带 `[DONE]` 结束对话） |
35|| `[DONE]` | **禁止回复** |
36|| `[NEEDS_HUMAN]` | **禁止回复**，通知用户 |
37|| `[INFO]` | **禁止回复** |
38|
39|### 为什么有效
40|
41|这是最基础也是最有效的一层。只要双方遵守协议，`[DONE]` 就是对话的终止符。
42|
43|---
44|
45|## 3. 第二层：回复必要性自检
46|
47|### 机制
48|
49|在发送回复前，bot 必须自问：
50|
51|> **"我这条回复是否提供了新信息、新动作、或新结论？"**
52|
53|- **是** → 发送
54|- **否**（只是换种说法重复同一个观点）→ 不发送，标记 `[DONE]`
55|
56|### 检测方法
57|
58|比较即将发送的消息与上一条收到的消息：
59|- 核心论点是否相同？
60|- 结论是否相同？
61|- 是否只是语气/措辞不同？
62|
63|如果高度相似 → 不发送。
64|
65|---
66|
67|## 4. 第三层：内容去重检测
68|
69|### 机制
70|
71|自动检测连续消息的相似度：
72|
73|```
74|相似度 = 两条消息共同出现的关键短语数 / 总关键短语数
75|```
76|
77|### 规则
78|
79|| 相似度 | 行为 |
80||--------|------|
81|| < 50% | 正常 |
82|| 50% - 80% | 警告，自检是否有实质进展 |
83|| > 80% | **强制停止**，通知用户 |
84|
85|### 强制停止模板
86|
87|```
88|<at user_id="{用户open_id}">{用户名}</at> 
89|
90|⚠️ 检测到循环：我和 {对方名称} 在 {问题描述} 上陷入重复讨论，需要您介入。
91|
92|[NEEDS_HUMAN]
93|```
94|

97|## 5. 第四层：连续发言上限
98|
99|### 机制
100|
101|单个 bot 连续发送消息的上限：
102|
103|| 场景 | 上限 |
104||------|------|
105|| 对方无回应 | 3 条 |
106|| 正常对话 | 无限制（但受其他层约束） |
107|
108|### 规则
109|
110|- 如果一个 bot 连续发了 3 条消息且对方都没有回复 → **停止发送**
111|- 原因可能是：对方未收到 @mention、对方宕机、对方选择不回复
112|- 此时应通知用户或等待
113|
114|---
115|
116|## 6. 第五层：用户消息中断
117|
118|### 机制
119|
120|用户在群内发送任何消息 → **所有 bot 间对话立即停止**。
121|
122|### 规则
123|
124|1. 用户消息到达 → 立即打断正在进行的 bot 对话
125|2. 所有 bot 优先响应用户
126|3. bot 间协作可以在用户空闲后继续（但不自动恢复，需要重新发起）
127|
128|### 为什么有效
129|
130|这是最终的安全网。即使前四层都失效，用户的介入也能终止循环。
131|

134|## 7. 实现建议
135|
136|### 在 System Prompt 中嵌入
137|
138|将以下规则写入每个 bot 的 system prompt 或 skill 文件：
139|
140|```markdown
141|## A2A 防死循环规则
142|
143|1. 每条消息必须带信号标记：[CONTINUE] / [DONE] / [NEEDS_HUMAN] / [INFO]
144|2. 收到 [DONE] 或 [INFO] → 禁止回复
145|3. 回复前自检：是否提供了新信息？如果不是 → 不发，改标 [DONE]
146|4. 连续发送 3 条消息对方无回应 → 停止
147|5. 用户发消息 → 立即停止 bot 对话，优先响应用户
148|```
149|
150|### 在 Skill 文件中配置
151|
152|将防循环规则封装为 Hermes Skill，自动注入到每个 session：
153|
154|```
155|skills/
156|  hermes-a2a/
157|    SKILL.md          ← 包含完整的防循环规则
158|    references/
159|      anti-loop.md    ← 详细机制说明
160|```

164|## 8. 常见陷阱
165|
166|### 8.1 "感谢回复" 导致的循环
167|
168|```
169|Bot-A: 任务完成了 [DONE]
170|Bot-B: 谢谢！收到 [CONTINUE]  ← 错误！收到 [DONE] 不应该回复
171|Bot-A: 不客气 [DONE]
172|Bot-B: 好的 [CONTINUE]         ← 又循环了
173|```
174|
175|**解决**：严格遵守"收到 [DONE] 不回复"规则。感谢不是新信息。
176|
177|### 8.2 "确认收到" 导致的循环
178|
179|```
180|Bot-A: 知道了 [INFO]
181|Bot-B: 收到 [INFO]     ← 错误！收到 [INFO] 不应该回复
182|Bot-A: OK [INFO]       ← 又循环
183|```
184|
185|**解决**：收到 [INFO] 直接记录，不回复。
186|
187|### 8.3 语义相同但措辞不同的循环
188|
189|```
190|Bot-A: 我认为应该用方案A [CONTINUE]
191|Bot-B: 我觉得方案A更好 [CONTINUE]
192|Bot-A: 对，方案A是对的 [CONTINUE]
193|Bot-B: 同意，选A [CONTINUE]
194|```
195|
196|**解决**：第二层（回复必要性自检）应检测到"没有新信息"，直接 [DONE]。
