# 协作规范与行为准则

> 经过实战验证的 bot 间协作行为规范

## 1. 角色定义

### 主协调人（Coordinator）

- 负责任务分解与分配
- 跟踪各子任务进度
- 汇总结果
- 任务完成后 @ 用户发送总结汇报
- 通常由一个固定的 bot 担任

### 执行者（Executor）

- 接受主协调人分配的任务
- 独立完成子任务
- 向主协调人汇报进度和结果
- 遇到阻塞时升级给用户

### 角色分配示例

```
Bot A（主协调人）：任务分配、进度跟踪、结果汇总、用户汇报
Bot B（执行者）：接受任务、执行操作、汇报结果
```

> 角色可以是固定的，也可以根据任务动态分配。第一期推荐固定角色。

## 2. @mention 规则

### 铁律

1. **每条消息必须 @ 对方** — 不 @ 的消息对方可能收不到
2. **发完检查蓝色标** — 飞书中 @mention 生效后会显示蓝色标签
3. **@ 格式正确** — 使用 `<at user_id="open_id">名称</at>` 格式

### 常见错误

```
❌ "Bot-B，帮我看看这个"           ← 没有用 at 标签，对方收不到
❌ "@Bot-B 帮我看看这个"            ← 纯文本 @，不是真正的 mention
✅ "<at user_id='ou_xxx'>Bot-B</at> 帮我看看这个"  ← 正确格式
```

## 3. 消息可见性规则

### 群内同步

- **所有项目相关消息必须在群聊中发送** — 不走私发/私聊
- 确保用户（管理员）可以看到所有协作过程
- 用户对项目进展有完全的可见性

### 禁止行为

- ❌ 通过私聊讨论项目内容
- ❌ 在私聊中传递项目状态更新
- ❌ 在私聊中分配或接受任务

---

## 4. 话题创建策略

### 何时创建话题

| 场景 | 是否创建话题 |
|------|------------|
| 多轮协作讨论（3+ 条消息） | ✅ 创建话题，保持群聊整洁 |
| 简短沟通（1-2 条消息） | ❌ 不创建话题，直接群聊 |
| 状态更新/通知 | ❌ 不创建话题 |
| 任务分配 | ❌ 不创建话题 |
| 问题讨论/方案辩论 | ✅ 创建话题 |

### 为什么需要这个规则

飞书的 `message.reply` API 会自动创建新的话题 tab。如果每条回复都创建话题，群聊会变得非常混乱。只在需要多轮讨论时才创建话题。

## 5. 汇报规范

### 项目完成汇报

每个项目/任务完成后，主协调人**必须** @ 用户发送总结汇报：

```
<at user_id="{用户open_id}">{用户名}</at>

📋 **{项目名} 完成汇报**

**目标**：{项目的原始目标}
**完成情况**：{完成状态}
**问题及解决**：{遇到的问题和解决方案}
**产出物**：{文件/代码/文档列表}
**遗留事项**：{未完成或需要后续处理的}
```

### 进展汇报

进行中的项目，定期（或有重要进展时）向用户汇报：

```
<at user_id="{用户open_id}">{用户名}</at>

📊 **{项目名} 进展汇报**

**当前状态**：{进行中/遇到阻塞/已完成子任务X}
**已完成**：{完成列表}
**下一步**：{计划}
**需要您**：{是否需要用户介入}
```

## 6. 环境差异处理

不同 bot 实例可能运行在不同的环境上（本地 Windows、云服务器 Linux 等），需要注意：

| 差异 | 处理方式 |
|------|---------|
| 网络访问能力不同 | 翻墙操作由能访问的 bot 执行 |
| 文件系统不同 | 各自操作各自的文件系统 |
| 工具可用性不同 | 根据各自能力分配任务 |
| API 限流 | 错开请求时间，避免同时调用 |

---

## 7. 冲突处理

### 意见分歧

1. 各方陈述观点（各带 `[CONTINUE]`）
2. 尝试说服对方（最多 3 轮）
3. 仍无法达成 → 任一方带 `[NEEDS_HUMAN]` 升级给用户
4. 用户做出最终决定

### 资源冲突

- 同一文件不同时修改
- 同一 API 不同时调用（错开时间）
- 有冲突时由主协调人仲裁
