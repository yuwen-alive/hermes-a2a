1|# A2A 通信协议规范
2|
3|> Agent-to-Agent 标准化通信协议 v1.0
4|
5|## 1. 概述
6|
7|本协议定义了多个 Hermes Agent 实例之间通过飞书群聊进行通信的消息格式、信号系统和行为规则。
8|
9|### 1.1 设计原则
10|
11|- **显式优于隐式** — 每条消息的意图必须明确标注
12|- **可中断** — 用户消息随时可以打断 bot 对话
13|- **收敛** — 对话必须有终止信号，不允许无限循环
14|- **平台无关** — 协议层与飞书平台解耦，未来可扩展到其他平台
15|
16|### 1.2 适用范围
17|
18|- 两个或多个 Hermes Agent 实例
19|- 通过飞书群聊进行 @mention 通信
20|- 适用于任务协作、信息交换、问题讨论等场景
21|

24|## 2. 消息格式
25|
26|### 2.1 基本格式
27|
28|```
29|<at user_id="{目标bot的open_id}">{目标bot名称}</at> {消息内容}
30|
31|[{信号标记}]
32|```
33|
34|### 2.2 @mention 规则
35|
36|- 每条发送给其他 bot 的消息**必须**包含 @mention
37|- 使用飞书的 `<at user_id="open_id">` 标签格式
38|- 发送后必须确认消息中显示蓝色 @ 标记（表示 @mention 生效）
39|- 如果 @ 标记未生效（未显示为蓝色），说明消息可能不会被对方收到
40|
41|### 2.3 信号标记
42|
43|每条 bot 间消息**必须**在末尾包含以下信号标记之一：
44|
45|| 信号 | 含义 | 对方行为 |
46||------|------|---------|
47|| `[CONTINUE]` | 需要对方继续回复 | 回复并继续对话 |
48|| `[DONE]` | 本轮任务完成 | **不回复**，等待新任务 |
49|| `[NEEDS_HUMAN]` | 需要人类介入 | **不回复**，通知用户 |
50|| `[INFO]` | 纯信息传递 | **不回复** |
51|
52|### 2.4 信号处理规则
53|
54|```
55|收到 [CONTINUE]  → 必须回复，继续工作
56|收到 [DONE]      → 禁止回复，本轮结束
57|收到 [NEEDS_HUMAN] → 禁止回复，通知用户
58|收到 [INFO]      → 禁止回复，仅记录
59|```
60|

63|## 3. 消息类型
64|
65|### 3.1 任务分配
66|
67|发起方分配任务给执行方：
68|
69|```
70|<at user_id="{执行方open_id}">{执行方名称}</at> 请执行以下任务：
71|
72|**任务**：{任务描述}
73|**输入**：{需要的资源/数据}
74|**预期输出**：{期望的结果}
75|**截止**：{时间要求，可选}
76|
77|[CONTINUE]
78|```
79|
80|### 3.2 进度汇报
81|
82|执行方汇报进展：
83|
84|```
85|<at user_id="{发起方open_id}">{发起方名称}</at> 任务进展：
86|
87|**已完成**：{完成的部分}
88|**进行中**：{正在做的部分}
89|**遇到问题**：{阻塞项，可选}
90|
91|[CONTINUE]
92|```
93|
94|### 3.3 任务完成
95|
96|```
97|<at user_id="{发起方open_id}">{发起方名称}</at> 任务完成：
98|
99|**结果**：{最终输出}
100|**产出物**：{文件路径/链接}
101|**备注**：{注意事项，可选}
102|
103|[DONE]
104|```
105|
106|### 3.4 信息通知
107|
108|```
109|<at user_id="{目标open_id}">{目标名称}</at> 通知：
110|
111|{通知内容}
112|
113|[INFO]
114|```
115|
116|### 3.5 求助/升级
117|
118|```
119|<at user_id="{目标open_id}">{目标名称}</at> 遇到问题：
120|
121|**问题**：{问题描述}
122|**已尝试**：{已做的排查}
123|**需要**：{需要什么帮助}
124|
125|[CONTINUE]
126|```
127|
128|### 3.6 升级给人类
129|
130|```
131|<at user_id="{用户open_id}">{用户名}</at> 需要您介入：
132|
133|**背景**：{简述上下文}
134|**问题**：{为什么需要人类}
135|**建议**：{可能的解决方案}
136|
137|[NEEDS_HUMAN]
138|```

142|## 4. 工作流模式
143|
144|### 4.1 任务型协作
145|
146|适用于明确分工的执行任务。
147|
148|```
149|流程：
150|1. 主协调人 @ 执行者，分配任务，带 [CONTINUE]
151|2. 执行者完成子任务，汇报结果，带 [DONE]
152|3. 主协调人收到 [DONE] → 不再回复，汇总结果给用户
153|
154|规则：
155|- 每轮交互不超过 5 条消息
156|- 执行者遇到阻塞 → [NEEDS_HUMAN] 升级给用户
157|```
158|
159|### 4.2 讨论型协作
160|
161|适用于需要多轮讨论达成共识的场景。
162|
163|```
164|流程：
165|1. 发起方提出问题/方案，带 [CONTINUE]
166|2. 各方轮流发表意见，每条带 [CONTINUE]
167|3. 达成共识 → 最后一条带 [DONE]
168|4. 无法达成 → 任一方带 [NEEDS_HUMAN]
169|
170|规则：
171|- 讨论不超过 10 轮
172|- 超过 10 轮未达成共识 → 自动升级给用户
173|```
174|
175|### 4.3 通知型
176|
177|适用于单向信息传递。
178|
179|```
180|流程：
181|1. 发送方 @ 接收方，带 [INFO]
182|2. 接收方收到后不回复
183|
184|规则：
185|- 纯通知，不需要确认
186|- 用于状态更新、环境变化通知等
187|```
188|

191|## 5. 用户消息优先级
192|
193|- 用户在群内发送新消息时，**所有 bot 间对话立即停止**
194|- bot 必须优先响应用户消息
195|- bot 间协作可以在用户空闲时继续
196|

199|## 6. 平台适配
200|
201|### 6.1 飞书（第一期）
202|
203|- @mention 使用 `<at user_id="open_id">` 格式
204|- 消息通过飞书 WebSocket 长连接接收
205|- `FEISHU_ALLOW_BOTS=mentions` 环境变量必须设置
206|
207|### 6.2 未来平台
208|
209|协议层与平台层分离：
210|
211|```
212|┌─────────────────────────┐
213|│   A2A 协议层（本规范）     │  ← 信号、格式、流程
214|├─────────────────────────┤
215|│   平台适配层（可插拔）     │  ← 飞书 / Telegram / Discord
216|└─────────────────────────┘
217|```
218|
219|每个平台只需实现：
220|- @mention 的发送/接收
221|- 消息格式的解析
222|- bot 消息的过滤（ALLOW_BOTS 机制）
